# Feature Specification: Session Summary Mode

**Feature Branch**: `007-session-summary-mode`  
**Created**: 2026-05-05  
**Status**: Draft  
**Input**: User description: "I want to add a new feature for hoosh. In this mode, every complete turn of the agent only retains conversation history, that is from user request to agent doing its ReAct thing until control is given back to user. Then agent should have a tool call update_session_file which is a summary of what was done (obviously agent needs to be prompted properly to do so) the idea is that we inject the summary session file generated by agent at the start of every turn. the hypothesis is that this creates less token consumption and is as good as retaining the full conversation history. We should develop this mode as seperate as possible and not touch existing code as much as possible"

## Clarifications

### Session 2026-05-05

- Q: How is memory mode activated — CLI flag only, both CLI flag and config field, or config only? → A: Both CLI flag (`--memory-mode summary|conversation`) and config field. Mode is an enum: `summary` (session summary mode) and `conversation` (existing full-history mode, the default).
- Q: Should `update_session_file` be a real registered tool, reuse `MessageSummarizer`, or be a pseudo-tool? → A: Real tool in the tool registry, available only in memory mode summary — agent explicitly calls it.
- Q: Where is the session summary file stored and does it survive process exit / `--continue`? → A: Dedicated path at `.hoosh/memory/<conv_id>/summary.txt`, written regardless of whether `conversation_storage` is enabled. Survives process exit and is available on `--continue`. Full conversation history and the summary file are independent concerns.
- Q: If agent skips `update_session_file`, what context does the next turn use? → A: Fall back to full raw history for that turn only — quality cannot be sacrificed. Summary mode resumes normally from the turn after.
- Q: Where and how is the session summary injected into context each turn? → A: As a `system` message in `actions.rs::answer()` (and tagged mode equivalent), after `add_user_message()` and before `handle_turn()` — once per turn, not via the system reminder infrastructure.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Enable Memory Mode: Summary (Priority: P1)

A user enables `--memory-mode summary` (or sets it in config) when starting a conversation. In this mode, at the end of each agent turn (from user request through all ReAct steps back to user control), the agent calls `update_session_file` with a concise summary of what was accomplished. At the start of the next turn, that summary is automatically injected into the conversation context, replacing the full prior history.

**Why this priority**: This is the core mechanism of the entire feature. Without it, nothing else can function.

**Independent Test**: Can be fully tested by starting a conversation with `--memory-mode summary`, completing one turn, and verifying that the next turn begins with the injected summary rather than the full raw history.

**Acceptance Scenarios**:

1. **Given** a user starts hoosh with `--memory-mode summary`, **When** the agent completes a full turn (user request → ReAct → response), **Then** the agent calls `update_session_file` with a structured summary of that turn before yielding control back to the user.
2. **Given** a session summary file exists from a prior turn, **When** the user sends the next message, **Then** the summary content is injected into the conversation context before the agent processes the new request.
3. **Given** memory mode summary is active and multiple turns have been completed, **When** the agent finishes the latest turn, **Then** the new summary overwrites the previous one so only the most recent summary is retained.

---

### User Story 2 - Reduced Token Consumption vs Full History (Priority: P2)

A user with long-running tasks benefits from significantly reduced token usage because only the current turn's messages plus the injected session summary are sent to the model, rather than the full accumulated conversation history.

**Why this priority**: The primary hypothesis driving the feature. Without observable token reduction, the mode offers no practical advantage over `--memory-mode conversation`.

**Independent Test**: Can be tested by running the same multi-turn task with `--memory-mode summary` and `--memory-mode conversation` and comparing total tokens consumed across all turns.

**Acceptance Scenarios**:

1. **Given** a conversation with 10+ prior turns in memory mode summary, **When** the user sends a new message, **Then** the context sent to the model contains only the session summary and the current turn's messages — not all prior raw messages.
2. **Given** memory mode summary is active, **When** the agent completes a turn, **Then** the generated summary is substantially shorter than the full raw message history it replaces.

---

### User Story 3 - Memory Mode: Conversation is Default and Unaffected (Priority: P3)

`--memory-mode conversation` (the default) behaves identically to the existing full-history mode. No existing behavior changes.

**Why this priority**: Ensures backward compatibility. The default must be unchanged.

**Independent Test**: Can be fully tested by running existing conversations without specifying `--memory-mode` and confirming behavior is identical to before the feature was introduced.

**Acceptance Scenarios**:

1. **Given** a user does not specify `--memory-mode`, **When** they have a conversation, **Then** the full conversation history is retained and used exactly as before (`conversation` is the default).
2. **Given** the memory mode code is present in the system, **When** a user runs in `conversation` mode, **Then** all existing tests pass without modification.

---

### Edge Cases

- If the agent fails to call `update_session_file` at the end of a turn: skip `clear_turn_history()` on the next turn, preserving full raw history silently. Summary injection resumes once the agent writes a summary again.
- How does the system handle the very first turn when no prior session summary file exists yet?
- What happens if the session summary file is corrupted, empty, or unreadable when the next turn starts?
- If the agent calls `update_session_file` mid-turn rather than at the end: multiple calls within a turn are idempotent (last write wins). The summary file is only read at turn start, so mid-turn writes have no effect on the current turn's context.
- What happens if the summary itself is so large that it approaches token limits?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide a `--memory-mode` flag (CLI) and corresponding config field accepting values `summary` and `conversation`.
- **FR-002**: `conversation` MUST be the default value for `--memory-mode`, preserving all existing full-history behavior.
- **FR-003**: When `--memory-mode summary` is active, the agent MUST be instructed (via system prompt or equivalent mechanism) to call `update_session_file` at the end of every complete turn with a structured, concise summary of that turn's work.
- **FR-003a**: `update_session_file` MUST be a real tool registered in the tool registry, made available to the agent only when `--memory-mode summary` is active.
- **FR-004**: The `update_session_file` tool MUST write the summary to `.hoosh/memory/<conv_id>/summary.txt`, overwriting any previous summary from a prior turn. This path is independent of the `conversation_storage` setting — the summary file is always persisted regardless of whether full conversation history storage is enabled. The file survives process restarts and is available when the session is resumed with `--continue`.
- **FR-005**: At the start of each new turn in memory mode summary, the system MUST inject a `system` message **before** `add_user_message()` is called and before `handle_turn()` runs — exactly once per turn, not via the system reminder infrastructure. The injected message always contains the agent's memory-mode instructions; on turn 2 and beyond it also contains the prior turn's summary.
- **FR-006**: The injected session summary MUST replace prior raw conversation history so that only the summary and the current turn's messages are part of the model's context window.
- **FR-007**: If no session summary file exists at the start of a turn (first turn of a session), the system MUST proceed without injection and treat it as a fresh start.
- **FR-007a**: If the agent does not call `update_session_file` during a turn, the system MUST fall back to skipping `clear_turn_history()` on the following turn, preserving full raw conversation history for that turn. Summary injection resumes normally once the agent writes a summary again.
- **FR-008**: If the session summary file is missing or unreadable at turn start, the system MUST log a warning and fall back to full raw history rather than failing the turn.
- **FR-009**: The implementation MUST be self-contained with minimal modification to existing conversation handling code, keeping new code in its own module or file(s).
- **FR-010**: `--memory-mode conversation` MUST NOT alter any existing behavior; all existing test suites pass without modification.
- **FR-011**: The agent's system prompt in memory mode summary MUST instruct it to summarize actions, outcomes, and relevant context — not raw message content — to keep summaries concise and useful.

### Key Entities

- **Session Summary File**: Stored at `.hoosh/memory/<conv_id>/summary.txt`. Contains the most recent turn's summary as structured text: what was accomplished, key decisions made, and relevant context. Written regardless of `conversation_storage` setting; persists across process restarts.
- **Turn**: A complete cycle from the user's request through all agent ReAct steps (reasoning, tool calls) until control is returned to the user.
- **Memory Mode**: An enum config/flag controlling how conversation history is managed. Values: `summary` (turn-by-turn summary injection) and `conversation` (full history retention, default).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: In a 10-turn conversation performing equivalent tasks, total tokens consumed in `--memory-mode summary` is at least 30% lower than in `--memory-mode conversation`.
- **SC-002**: Agent task completion accuracy in memory mode summary is equivalent to conversation mode for standard multi-step tasks, with no measurable regression in correctness.
- **SC-003**: The session summary file is written before control returns to the user on every completed turn under normal operation — zero missed writes.
- **SC-004**: Enabling memory mode summary requires no changes to the existing user workflow beyond specifying `--memory-mode summary` as a flag or config entry.
- **SC-005**: All existing test suites pass without modification when `--memory-mode conversation` (default) is used.

## Assumptions

- The agent is capable of producing useful, concise summaries when properly prompted; no additional model capability beyond what hoosh already uses is required.
- A "turn" boundary is well-defined — it corresponds to `Agent::handle_turn()` returning in `src/agent/core.rs`.
- Session identity for naming the summary file can be derived from existing session or conversation identifiers already present in the system.
- The session summary file is stored at `.hoosh/memory/<conv_id>/summary.txt`, independent of `conversation_storage`. This is a separate, lightweight storage concern from full conversation history.
- Only the most recent summary is relevant; the file is overwritten each turn, not appended.
- The initial implementation does not need to support concurrent multi-session scenarios.
